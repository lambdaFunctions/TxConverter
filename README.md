# TxConverter

A FastAPI service that stores purchase transactions in USD and converts them to a target currency using the US Treasury Rates of Exchange API.

---

## Prerequisites

- Docker (for pipeline and production)

---

## Starting the project

```bash
bash pipeline.sh
```

The pipeline file will ensure the build of test and production images, the execution
of the tests and coverage report and then execute the production application.

Swagger interface is available at `localhost:8000/docs`

## Architecture

### Architectural Decisions

The project architecture chosen was Layered Architecture. The software architecture chosen was Decentralised Coordination.

### Local File For Transactions

- Core ideas:
    - Avoid using SQLite
    - Prove multiple write and read from single source in parallel and concurrent context
    - Not using append-only approach, which causes the entire file to be loaded into
      memory in every write/update/delete, but it allows the API to create the
    complete CRUD methods if necessary.

### In Memory Cache For The Currencies

A disaster recovery focused solution would write the cache into a file and handle the TTL for the registers to avoid letting the
file grow too much and yet deliver the content once the server is restarted.
However, the proves of file-handling techiniques are shown in the `transactions`
implementation. Furthermore, the currency related data can be replicated by calling
an external API while the `transactions` should be persisted given that this data
comes from a non-controlled environment: the client. This cannot be replicated
accordingly to our will.

Choosing to have in-memory cache for `currencies` has a bias of showing my hability
of the dev while accepting the IO latency as a trade-off for the case.

### Project Structure

Each layer has a single responsibility:

- **Router** — declares the HTTP contract: status codes, request/response schemas, and all documented error codes via `responses={}`
- **Use Case** — orchestrates the flow for one endpoint; calls services, raises domain exceptions, maps models to response schemas
- **Service** — pure domain logic; no HTTP knowledge; handles computation (amounts, date math, conversion)
- **Repository** — reads and writes the transaction file; no domain logic
- **External Client** (`TreasuryClient`) — wraps all HTTP calls to the Treasury API; translates transport errors into `InfraFetchCurrencyError`

**Error handling** is uniform across the entire API:

- All business exceptions extend `TxConverterBaseException` with a fixed `http_status`, `code`, and `message`
- `ExceptionHandlerMiddleware` catches every exception, logs it, and calls `build_error_response` for known types or `build_fallback_response` for unknown ones
- Every error response has the same shape: `{"error": {"code": "...", "message": "..."}}`

## Persistence

### Transaction store

- A single flat JSON file at the path configured by `TRANSACTIONS_FILE_PATH`
- Format: `{ "<id>": { transaction fields } }`
- **Writes**: read current file → mutate in memory → write to a `.tmp` file → atomically rename to the real path with `os.replace`. This guarantees the file is never in a partial or corrupt state even if the process crashes mid-write
- **Reads for listing**: streamed with `ijson.kvitems` — the file is never fully loaded into memory; the iterator filters and paginates in a single pass with O(page_size) peak memory
- **Reads for lookup**: `json.load` into a dict, then key lookup by ID
- **File size cap**: enforced before every write via `TX_FILE_MAX_BYTES`; returns HTTP 507 if exceeded

### Exchange rate cache

- A flat JSON file in `CACHE_DIR/rates.json`
- Format: `{ "<country_currency_desc>": { "value": { "<date>": <record> }, "ts": <unix timestamp> } }`
- Entries expire after 24 hours (TTL); expired keys are evicted on every write
- **Writes** use `portalocker` (cross-process exclusive file lock) to prevent concurrent write corruption
- **Reads** are lock-free; atomic `os.replace` writes ensure the file is always in a valid JSON state when read
- **Cache file size cap** enforced before every write via `CACHE_FILE_MAX_BYTES`

### Transaction IDs

- Generated by **SonyFlake** — a distributed snowflake-style ID algorithm
- Each node is identified by `MACHINE_ID`, ensuring IDs are unique across multiple containers without coordination

## Concurrency and parallelism

This application is designed to be safe under both concurrent async tasks (within one process) and parallel OS processes (multiple uvicorn workers).

### Multiple OS processes (true parallelism)

- `entrypoint.sh` starts uvicorn with `--workers $(nproc)`, creating one worker process per CPU core
- Each process has its own Python GIL, memory space, and event loop — they run fully in parallel
- Shared state lives only in files (`transactions.json`, `rates.json`), not in process memory

### Async I/O (concurrency within each worker)

- All route handlers are `async def` — FastAPI runs them on the event loop without blocking
- `httpx.AsyncClient` is used for all Treasury API calls — the event loop is never blocked waiting for a network response
- File I/O is synchronous by nature (the OS has no async file API at the kernel level), so all file reads and writes are offloaded with `asyncio.to_thread`, which runs them in a thread pool and yields control back to the event loop while they execute

```
Event loop
  ├─ request A: awaits httpx Treasury call (non-blocking)
  ├─ request B: awaits asyncio.to_thread(file write) (non-blocking to the loop)
  └─ request C: running route handler logic (sync, fast path)
```

### File safety across processes

- **Transaction writes**: shared-lock on read (`fcntl.LOCK_SH`), then atomic `os.replace` — multiple workers can read simultaneously; the write is invisible to other readers until the rename completes
- **Cache writes**: exclusive lock via `portalocker` (`LOCK_EX`) before reading and writing the cache file — only one worker at a time modifies the cache; reads are always lock-free against a consistent snapshot

### Logging safety

- Loguru is configured with `enqueue=True` — log records are placed on an internal queue and written by a dedicated background thread
- This means log calls from concurrent async tasks and multiple threads never block each other and never interleave output

### Correlation ID propagation

- Each request is assigned a unique ID by `CorrelationIdMiddleware` before any other processing
- The ID is stored in a `contextvars.ContextVar` — a per-async-task variable that does not bleed between concurrent requests
- The loguru filter reads the `ContextVar` before enqueuing the record, so the correct ID is captured in the log entry even though the actual write happens on a background thread later
